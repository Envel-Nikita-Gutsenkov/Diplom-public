# Архитектура песочниц выполнения кода Python (Python Sandbox Engine)

В системе Diplom-public реализовано два независимых механизма (песочницы) для интерпретации и тестирования решений на языке Python: клиентский (в браузере администратора/ученика) и серверный (в Docker-контейнере приложения при отправке решения).

---

## Сравнительная таблица сред выполнения

| Критерий | Клиентская песочница (Client-side Sandbox) | Серверная песочница (Server-side Sandbox) |
| :--- | :--- | :--- |
| **Технология** | WebAssembly (порт **Pyodide** v0.29.3 на базе CPython) | Нативный **Python 3.x** в Alpine Linux |
| **Место выполнения** | Фоновый поток браузера (Web Worker) | Серверный процесс (контейнер `app`) |
| **Где используется** | Конструктор тестов (админка), отладчик решений (`StudentCodeRunner`), песочница ученика | Окончательная проверка и оценка при сдаче олимпиады |
| **Изоляция ресурсов** | На уровне возможностей браузера (Web Worker) | Ограничение памяти до 32 МБ (`resource.setrlimit`) |
| **Поддержка библиотек** | Динамическая установка через `micropip` | Предустановленные пакеты в образе (или pip) |
| **Ограничение времени** | Сторожевой таймер браузера | Принудительное завершение (`SIGKILL`) через 5 сек |

---

## 1. Клиентская песочница (Pyodide & Web Worker)

Используется для моментального интерактивного тестирования и редактирования задач без отправки запросов на сервер в рамках системы Diplom-public.

### Компоненты взаимодействия
*   **Скрипт воркера**: [`public/workers/python-worker.js`](../public/workers/python-worker.js) загружает ядро Pyodide из CDN `jsdelivr`.
*   **Координаторы**:
    *   [`CodeEditor.tsx`](../components/admin/test-editor/CodeEditor.tsx) (конструктор тестов администратора).
    *   [`StudentCodeRunner.tsx`](../components/admin/StudentCodeRunner.tsx) (панель интерактивного разбора решений).
    *   [`PythonShell.tsx`](../components/tasks/PythonShell.tsx) (оболочка для выполнения кода учеником).

### Обработка I/O (ввода-вывода)
*   **stdout / stderr**: Перехватываются через методы `pyodide.setStdout` / `pyodide.setStderr` и отправляются в виде текстовых чанков обратно в UI.
*   **stdin**: Поскольку выполнение кода в воркере асинхронное, стандартная функция `input()` переопределяется через прокси, который блокирует поток воркера с помощью `Atomics.wait` и ждет заполнения `SharedArrayBuffer` данными из основного потока интерфейса.

## 2. Серверная песочница (Native Python & Security Wrapper)

Запускается на сервере при окончательной отправке решения учеником. Размещена в модуле управления оценками [`lib/grading/code-executor.ts`](../lib/grading/code-executor.ts).

### Установка Python на сервере
Среда выполнения Python вшита непосредственно в Docker-образ Next.js приложения Diplom-public. 

В файле [`Dockerfile`](../Dockerfile) используется базовая система `node:24-alpine`, поверх которой ставится системный пакет Python 3:
```dockerfile
FROM node:24-alpine AS base
RUN apk add --no-cache libc6-compat python3 py3-pip
```
Версия интерпретатора соответствует дистрибутиву Alpine Linux (обычно **Python 3.11** или **3.12**).

### Меры безопасности выполнения кода (Security Sandbox)
Поскольку код запускается локально внутри контейнера `app` через `child_process.spawn`, движок оборачивает код ученика в многоуровневую систему безопасности:

1.  **Ограничение памяти (Memory Limit)**
    На Linux-системах с помощью встроенной библиотеки `resource` устанавливается жесткий лимит в 32 МБ виртуальной памяти:
    ```python
    import resource
    max_mem = 32 * 1024 * 1024
    resource.setrlimit(resource.RLIMIT_AS, (max_mem, max_mem))
    ```
2.  **Запрет импорта опасных библиотек**
    Встроенный механизм загрузки модулей `__import__` переопределяется. При попытке импортировать системные утилиты или модули работы с сетью генерируется `ImportError`:
    ```python
    forbidden = ['os', 'sys', 'subprocess', 'shutil', 'socket', 'pathlib', 'pty', 'resource']
    ```
3.  **Блокировка доступа к файловой системе и динамического исполнения**
    Заглушаются встроенные функции:
    *   `open` -> выбрасывает `PermissionError("Работа с файлами запрещена")`
    *   `eval` -> выбрасывает `PermissionError("eval() запрещен")`
    *   `exec` -> выбрасывает `PermissionError("exec() запрещен")`
4.  **Запрет обхода через хитрый выход (Anti-Forgery Verification)**
    Для предотвращения ситуации, когда код ученика перехватывает управление и имитирует успешное прохождение тестов (например, досрочно вызывая `sys.exit(0)`), движок генерирует случайный UUID-токен (`secretToken`), который добавляется в самый конец кода.
    
    Тест признается успешным только в том случае, если процесс завершился с кодом возврата `0` **и** в выводе консоли строго присутствует этот UUID.
5.  **Контроль времени (Watchdog)**
    На выполнение одного тест-кейса отводится максимум 5000 мс. При превышении лимита Node.js принудительно убивает процесс командой `child.kill('SIGKILL')`.
