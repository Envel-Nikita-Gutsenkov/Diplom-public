# Автоматическое Тестирование и Оценивание Решений

Данный документ описывает подсистему автоматического тестирования и оценки ответов участников олимпиад. Архитектура подсистемы спроектирована для обеспечения мгновенной обратной связи, безопасности исполнения кода и гибкого оценивания как простых вопросов, так и задач по программированию.

---

## 1. Типы автоматического тестирования

В зависимости от типа задачи система применяет различные методы автоматического тестирования решений:

| Тип задачи | Метод автоматического тестирования | Критерий успешности прохождения |
| :--- | :--- | :--- |
| **Одиночный выбор (Single Choice)** | Сравнение выбранного ID варианта с эталонным. | Точное совпадение идентификатора. |
| **Множественный выбор (Multiple Choice)** | Пересечение множеств выбранных вариантов с эталонными. | Полное совпадение выбранного набора (без лишних и недостающих). |
| **Краткий текстовый/числовой ответ** | Строковое или математическое сопоставление ответа. | Совпадение строки (с учетом/без учета регистра, обрезки пробелов). |
| **Программирование (Python Task)** | Запуск кода в изолированной песочнице на наборе тест-кейсов. | Соответствие вывода (`stdout`) эталонному для каждого `stdin`. |

---

## 2. Архитектура тестирования программного кода (Python Sandbox Engine)

Тестирование задач по программированию является наиболее сложным процессом, так как требует безопасного выполнения произвольного кода участников. Для этого используется клиентский движок на базе WebAssembly (`Pyodide`) и фоновых потоков (`Web Workers`).

### Схема взаимодействия компонентов при тестировании кода:

```mermaid
graph TD
    UI[Интерфейс участника / админа]
    Executor[code-executor.ts]
    Worker[python-worker.js (Web Worker)]
    Pyodide[Pyodide Engine (WASM)]
    
    UI -->|Исходный код + Тест-кейсы| Executor
    Executor -->|Асинхронный запуск| Worker
    Worker -->|Изолированное выполнение| Pyodide
    Pyodide -->|Поток вывода stdout / stderr| Worker
    Worker -->|Результаты тестов| Executor
    Executor -->|Статус прохождения и баллы| UI
```

### Основные компоненты подсистемы:
1.  **Конструктор тест-кейсов (`CodeEditor.tsx`)**: Интерфейс администратора для привязки к задаче набора тестов. Каждый тест-кейс состоит из:
    *   `input` (входной поток данных, подаваемый в `stdin`).
    *   `output` (ожидаемый эталонный вывод из `stdout`).
    *   `isPublic` (флаг видимости теста для участника — например, для демонстрационных примеров).
2.  **Фоновый поток выполнения (`python-worker.js`)**: Выполняет код в изолированной среде Web Worker. Это гарантирует, что выполнение пользовательского кода (даже содержащего бесконечный цикл `while True`) не блокирует интерфейс браузера.
3.  **Координатор выполнения (`code-executor.ts`)**: Отвечает за последовательный прогон решения по всем тест-кейс сценариям, замер времени выполнения и подсчет успешных попыток.

## 3. Механизм выполнения и сопоставления результатов (Grading Pipeline)

Процесс тестирования одной отправки (Submission) кода состоит из следующих этапов:

1.  **Инициализация виртуальной машины**: Загрузка среды Pyodide, настройка встроенных модулей (например, `sys`, `io`) для перехвата ввода-вывода.
2.  **Перенаправление потоков (`I/O Redirection`)**:
    *   **Входной поток (`stdin`)**: Реализуется через подмену встроенной функции `input()` в Python, которая последовательно возвращает строки из предоставленного `input` тест-кейса.
    *   **Выходной поток (`stdout`)**: Перехватывается системный вывод `sys.stdout` в строковый буфер.
3.  **Исполнение и контроль времени (Watchdog)**:
    *   Код запускается асинхронно.
    *   Устанавливается сторожевой таймер. Если выполнение теста длится дольше лимита времени (например, 2-5 секунд на тест или 30 секунд на всю сессию), процесс принудительно прерывается со статусом `Timeout`.
4.  **Нормализация и сравнение результатов**:
    *   Полученный `stdout` очищается от хвостовых пробелов, непечатных символов и пустых строк в конце.
    *   Производится сравнение с эталонным `output` (регистрозависимое или регистронезависимое в зависимости от настроек).
5.  **Классификация исходов**:
    *   `PASSED` — Тест успешно пройден, вывод совпал с эталоном.
    *   `WRONG ANSWER` — Код завершился успешно, но вывод отличается от эталона.
    *   `RUNTIME ERROR` — Во время выполнения возникло исключение Python (например, `ZeroDivisionError`, `IndexError`), выводится текст ошибки (`stderr`).
    *   `SYNTAX ERROR` — Ошибка компиляции кода (невалидный синтаксис Python).
    *   `TIMEOUT` — Превышен лимит времени выполнения.

## 4. Алгоритм подсчета баллов (Scoring Algorithm)

Итоговая оценка за задачу по программированию вычисляется динамически на основе доли успешных тестов:

$$\text{Итоговый балл} = \text{Максимальный балл за задачу} \times \left( \frac{\text{Количество успешно пройденных тестов}}{\text{Общее количество тест-кейсов}} \right)$$

*Пример*: Если задача оценивается в 10 баллов, а решение успешно прошло 8 тестов из 10, участнику автоматически начисляется 8 баллов.

## 5. Интерактивная отладка администратором (`StudentCodeRunner`)

Для удобства проверки спорных решений и обработки апелляций в панель администратора интегрирован инструмент **StudentCodeRunner** (`StudentCodeRunner.tsx`).

### Возможности интерактивного тестирования:
*   **Тестирование на лету**: Запуск присланного участником решения прямо из карточки ответа в один клик.
*   **Кастомный `stdin`**: Возможность ввести любые произвольные входные данные в текстовое поле и посмотреть, как программа поведет себя на специфичных крайних случаях (edge cases).
*   **Детальный дифференциальный лог (Diff Log)**: Визуальное сравнение ожидаемого вывода и реального вывода программы с подсветкой несовпадающих символов/строк.
*   **Переопределение результатов**: По результатам ручного тестирования администратор может через панель `ManualGrading` изменить финальный балл за задачу, добавив текстовый комментарий для участника.
